Orgue IAMS · Présentation Direction

Comparatif Planning
Ancien site vs Nouveau prototype

Objectif de cette présentation : montrer les améliorations concrètes déjà actives, rassurer sur la sécurité et préparer une validation SI sereine.

Contexte Le nouveau planning conserve les usages essentiels et corrige les points devenus fragiles avec l’obsolescence de l’ancien site.

Important Le prototype est fonctionnel aujourd’hui, et ses réglages restent adaptables selon les retours de la Direction et du SI.

Navigation : ← → ou Espace

1 · Vue d’ensemble

Architecture générale du planning

Schéma simple : navigateur utilisateur → application planning (web) → services backend sécurisés.

Bloc Rôle
Application web (planning) Interface utilisateur : consultation, réservation, édition, administration.
Backend Supabase Authentification, base de données, contrôle d’accès, fonctions serveur.
Intégrations externes Consultation via Google Calendar et notifications e-mail selon paramétrage.

Point clé : l’application est la référence métier ; les agendas Google servent principalement à la consultation.

2 · Technologies

Technos mises en oeuvre (prototype actuel)

Front

  • HTML / CSS / JavaScript (modules ES)
  • FullCalendar (grille planning)
  • Tailwind CSS + DaisyUI (interface)
  • PWA (usage web fluide, mobile/tablette)

Backend

  • Supabase Auth (sessions et rôles)
  • PostgreSQL (données planning et profils)
  • Edge Functions Supabase (logique serveur)
  • Intégrations : Google Calendar, e-mail

Ces choix peuvent évoluer après revue SI, sans remettre en cause les besoins fonctionnels.

3 · Hébergement actuel

Où tourne le prototype aujourd’hui

Composant Hébergement actuel Remarque
Site web planning (front) GitHub Pages Site statique public (HTTPS), facile à démontrer.
Base de données Supabase (PostgreSQL) Données utilisateurs, créneaux, règles, statistiques.
Fonctions serveur Supabase Edge Functions Actions administratives, logique métier, intégrations.

Suite logique Le SI pourra valider ou ajuster cette cible d’hébergement selon ses standards de sécurité, RGPD et exploitation.

1 · Fonctionnel utilisateur

Ce qui change au quotidien

Nouveau planning

  • Édition des créneaux plus fluide (déplacement/redimensionnement).
  • Expérience mobile/tablette proche d’un agenda moderne.
  • Types de créneaux clairs : Travail, Cours, Concert, Autre, Fermeture.
  • Consultation Google Calendar (planning global + planning personnel).

Ancien planning

  • Vue semaine surtout, ergonomie web classique (non mobile).
  • Fonctions historiques dégradées avec le temps (titres, mails, stats).
  • Pas d’intégration Google calendar structurée pour la consultation.
  • Maintenance devenue difficile en pratique.
4 · Pédagogie

Cours, élèves inscrits, semaines A/B

Nouveau planning

  • Type Cours distinct des réservations simples.
  • Association cours ↔ élèves : meilleure lisibilité pédagogique.
  • Gestion des semaines A/B pour réduire les confusions.
  • Suivi plus fin des activités (cours et travail perso).

Ancien planning

  • Pas de structure cours/élèves.
  • Intitulés parfois insuffisants (ex. “Cour”).
  • Pas de logique semaine A/B.
  • Risque d’erreurs d’interprétation côté élèves.

Impact direction : amélioration de la clarté organisationnelle sans alourdir l’usage.

5 · Équité de réservation

Règles actives et paramétrables

Actif maintenant Quota élève travail perso : 2h/semaine (modifiable)

Actif maintenant Fenêtre max de réservation élève : 15 jours (modifiable)

Actif maintenant Contrôle de suppression pour limiter les abus

Actif maintenant Anti-chevauchement des réservations

Dans l’ancien site, ces règles étaient surtout orales : le nouveau les rend applicables, contrôlables et ajustables.

6 · Sécurité (message simple)

Pourquoi le nouveau socle est plus sûr

  • HTTPS : présent avant et maintenant (base commune).
  • Authentification modernisée : session/token, moins de logique “maison” fragile.
  • Droits renforcés : contrôles côté serveur + politiques d’accès en base.
  • Administration encadrée : rôles, suspension, gestion mots de passe.
  • Gouvernance : versionning GitHub, suivi des corrections/évolutions.

À cadrer avec le SI Politique de mot de passe, seuils d’échec, durées de session, conformité RGPD détaillée.

7 · Statistiques

Indicateurs déjà en place

  • Occupation globale sur période : Cours / Travail perso (élèves) / Concert / Total
  • Par élève : nombre de créneaux, heures, moyenne heures/semaine
  • Graphique journalier des heures de travail perso

Évolutif D’autres indicateurs peuvent être ajoutés selon les besoins de pilotage direction/SI.

Ancien site : module stats historique devenu non fonctionnel (API obsolètes).
