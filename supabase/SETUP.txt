Démarrage option 2 — Supabase + Google Agenda
=============================================

1) Projet Supabase
   - Créer un projet sur https://supabase.com
   - Authentication → activer Email (mot de passe)
   - SQL Editor → coller et exécuter le fichier schema.sql
   - Créer 1 compte test via Authentication → Users → Add user
   - Table Editor → profiles : ajuster display_name et role (admin / prof / eleve / consultation) pour ce compte

2) SQL complémentaire (règles + annonces + trigger invite)
   - SQL Editor → exécuter supabase/migrations/002_admin_content.sql
   - Si la base existe déjà, exécuter aussi supabase/migrations/003_consultation_role.sql
     puis supabase/migrations/004_reservation_types.sql, 005_planning_admin_list_users_rpc.sql,
     006_consignes_ict.sql (si besoin), 007_google_calendar_pool.sql (pool calendriers perso + affectation auto),
     009_organ_week_cycle.sql (semaines A/B affichées dans la barre du calendrier, réglage prof/admin : menu « Semaines types A / B »).
     010_organ_school_template.sql (année scolaire + plage chapelle, gabarit A/B, RPC élèves actifs + calendrier pool).
   - Pour créer les comptes de base demandés (admin/élèves/prof), exécuter supabase/seed-users.sql
     (le script crée bien auth.users + auth.identities + profiles ; ne pas recopier d’ancienne variante
     qui n’insérait que auth.users — cela cassait GET /auth/v1/admin/users / « Gestion des comptes ».)
   - Si la base a encore des utilisateurs sans identités (erreur admin liste) : exécuter une fois
     supabase/fix-missing-auth-identities.sql puis retester l’API admin.
   - Si l’API admin liste échoue encore : exécuter supabase/fix-auth-users-instance-id.sql
     (anciens seeds utilisaient instance_id = 00000000-… au lieu de auth.instances.id).
   - Si auth.instances est vide (0 lignes) et tous les instance_id restent 0000… (y compris après
     Add user) : exécuter une fois supabase/bootstrap-auth-instance.sql (voir en-tête du fichier),
     puis retester curl /auth/v1/admin/users.
   - Si la connexion échoue après seed SQL (identifiants invalides) : exécuter
     supabase/fix-auth-login-tokens-and-passwords.sql ou rejouer seed-users.sql à jour ;
     en dernier recours : Dashboard → Users → définir le mot de passe (hash généré par GoTrue).

2bis) Reprise à zéro (prototype)
   - Option A — même projet : Database → (si besoin) supprimer les données métier dans public.* ;
     pour Auth, supprimer les utilisateurs via Authentication → Users, ou exécuter avec prudence un
     script de nettoyage ; puis rejouer schema.sql + migrations + seed corrigé.
   - Option B — nouveau projet : créer un projet neuf, noter ref + clé anon, supabase link, db push
     des migrations, redeploy functions, recopier les secrets (GOOGLE_*, BREVO, etc.), puis mettre à jour
     js/config/planning.config.js (supabaseUrl, supabaseAnonKey, calendarBridgeUrl).

2ter) Créer plusieurs calendriers secondaires (pool, ex. 60)
   - L’agent ne peut pas créer des agendas sur votre compte Google : utilisez le script Apps Script
     `scripts/google-create-planning-calendars.gs` (script.google.com, connecté en orgue.iams@… ou compte
     propriétaire des calendriers).
   - Activer le service avancé « Google Calendar API » dans l’éditeur Apps Script (icône puzzle) pour le
     partage public en lecture (ACL) sur tous les « Planning IAMS NN ».
   - Lot supplémentaire 28–47 (20 calendriers) + partage public sur tous les IAMS + SQL pour Supabase :
     exécuter une fois la fonction `runOnce_CreatePlanning28to47_PublicAndPoolSql()`. Le journal et un
     fichier `.sql` sur Drive contiennent l’INSERT pour `google_calendar_pool` ; l’agent peut ensuite
     l’appliquer avec `npx supabase db query --linked -f …` si le fichier est dans le dépôt.
   - Si les calendriers 28–47 existent déjà mais sans visibilité publique : `runOnce_OnlyMakeAllPlanningIamsPublicReader()`.
   - Google limite souvent ~25–30 créations d’affilée : en cas d’erreur, relancer le même jour plus tard
     ou diviser la plage (modifier START_NUM / END_NUM dans le script pour d’autres lots).
   - Pour les 26 premiers dans Supabase : `supabase/manual/seed-planning-iams-pool-26.sql` (ou export Apps Script).
   - Miroir agenda perso (Edge calendar-bridge) : chaque calendrier du pool doit être partagé avec le
     même compte Google que pour GOOGLE_CALENDAR_ID (e-mail du compte de service ou compte OAuth),
     droit « Modifier les événements ». Sinon l’API renvoie souvent « Not Found » ou « writer access » sur le secondaire.
     Si GOOGLE_SERVICE_ACCOUNT_JSON est défini, le bridge utilise le SA (`client_email` du JSON), pas orgue.iams@… :
     partager chaque agenda « Planning IAMS xx » avec cet e-mail SA en « Modifier les événements » (le partage
     uniquement avec orgue.iams ne suffit pas).
     En base, stocker l’ID `xxx@group.calendar.google.com` (pas l’URL embed entière) — l’admin et le bridge
     normalisent désormais les URLs `?src=…`.
   - Contrôle du pool : `npx supabase db query --linked -f supabase/manual/verify-google-calendar-pool.sql -o table --agent=no`
     (ou SQL Editor si la CLI n’est pas liée).

3) Edge Functions
   - Installer la CLI : https://supabase.com/docs/guides/cli
   - supabase login && supabase link --project-ref <VOTRE_REF>
   - Déploiement local tout-en-un (migrations + toutes les fonctions) : à la racine du dépôt Planning, `npm run deploy:supabase`
     (lit la ref dans `supabase/project-ref`, ou `SUPABASE_PROJECT_REF` si défini ; même ordre que la CI : db push puis functions deploy).
   - Secrets du projet (Dashboard → Edge Functions → Secrets, ou CLI) pour Google Calendar API :
       GOOGLE_CALENDAR_ID = orgue.iams@google.com (ID du calendrier affiché dans les paramètres Google Agenda)
       Option A — compte de service (recommandé une fois le calendrier partagé avec l’e-mail du SA) :
         GOOGLE_SERVICE_ACCOUNT_JSON = contenu entier du fichier JSON de clé GCP (une seule ligne ou colle multiligne dans le dashboard)
       Option B — OAuth (refresh token du compte orgue.iams@google.com, obtenu une fois hors Edge) :
         GOOGLE_OAUTH_CLIENT_ID, GOOGLE_OAUTH_CLIENT_SECRET, GOOGLE_OAUTH_REFRESH_TOKEN
       Optionnel : GOOGLE_CALENDAR_TIMEZONE = Europe/Paris (défaut Europe/Paris)
       Workspace + délégation domain-wide : GOOGLE_CALENDAR_IMPERSONATE = orgue.iams@google.com
         et GOOGLE_CALENDAR_ID = primary (souvent le plus simple pour l’agenda principal).
   - supabase functions deploy calendar-bridge
   - supabase functions deploy planning-admin
   - supabase functions deploy planning-slot-notify
   (La secrétaire a le rôle profiles.role = admin ; les fonctions « gestion comptes » passent par planning-admin.)
   - E-mails « créneau modifié par un tiers » (Brevo API, distinct du SMTP Auth Supabase) :
       BREVO_API_KEY — clé SMTP & API (Brevo → SMTP & API)
       NOTIFY_FROM_EMAIL — expéditeur vérifié sur le domaine Brevo (ex. no-reply@…)
       NOTIFY_FROM_NAME — optionnel (défaut : Planning Orgue IAMS)

4) Google Cloud (API Calendar, sans Apps Script)
   - Console Google Cloud → activer l’API « Google Calendar » pour le projet.
   - Compte de service → clé JSON → copier dans le secret GOOGLE_SERVICE_ACCOUNT_JSON.
   - Dans Google Agenda (compte orgue.iams@google.com) : Paramètres du calendrier →
     Partager avec l’adresse e-mail du compte de service (autorisation « Modifier les événements »),
     sauf si vous utilisez la délégation domain-wide (Workspace) avec GOOGLE_CALENDAR_IMPERSONATE.
   - Scopes OAuth si vous passez par refresh token : https://www.googleapis.com/auth/calendar

5) Front (Planning)
   - js/config/planning.config.js :
       supabaseUrl = https://<ref>.supabase.co
       supabaseAnonKey = clé anon (Settings → API)
       calendarBridgeUrl = https://<ref>.supabase.co/functions/v1/calendar-bridge

6) Auth : URLs de redirection
   - Supabase → Authentication → URL Configuration
   - Site URL = origine exacte du front (sans chemin en trop si ton app est à la racine du déploiement)
   - Redirect URLs : idem + variantes (localhost pour le dev, prod pour la prod)
   - Exemple GitHub Pages (site statique, sans hébergeur « classique ») :
       https://<user>.github.io/<repo>/
     ou, si tu ajoutes un nom de domaine perso sur Pages : https://planning.iams.fr/
   - Chaque origine utilisée (connexion, lien dans un e-mail) doit être listée dans Redirect URLs.

6bis) Pas d’hébergeur web : normal
   - GitHub Pages sert uniquement les fichiers HTML/JS/CSS ; ce n’est pas là que tu configures le SMTP.
   - Brevo n’a pas besoin de ton « hébergeur » : il a besoin d’un expéditeur autorisé (voir §7).
   - Pour un e-mail pro (no-reply@…) sans serveur web : un nom de domaine chez un **registrar** (Gandi, OVH, Cloudflare, etc.) suffit pour poser les enregistrements DNS (SPF/DKIM) vers Brevo — pas d’offre d’hébergement obligatoire.
   - Sans domaine du tout : tu peux souvent utiliser une adresse d’expéditeur **vérifiée** chez Brevo (ex. ton Gmail perso en expéditeur de test) ; moins idéal pour l’image IAMS mais fonctionnel pour les tests.

Note : si execute function dans schema.sql échoue, remplacer par :
   for each row execute procedure public.handle_new_user();

7) E-mails (Brevo → Supabase SMTP)
   - Compte sur https://www.brevo.com — vérifier un domaine d’envoi (ex. iams.fr) : Brevo donne les enregistrements DNS (SPF, DKIM) à ajouter chez ton hébergeur DNS.
   - Brevo → SMTP & API : créer une clé SMTP (identifiant + mot de passe dédiés au relais).
   - Supabase Dashboard → Project Settings → Authentication (ou Emails) → SMTP Settings :
       Host = smtp-relay.brevo.com
       Port = 587 (TLS)
       User = identifiant SMTP Brevo (souvent ton email de compte ou une clé dédiée, selon l’écran)
       Password = clé SMTP
       Sender email = adresse autorisée sur le domaine vérifié (ex. no-reply@tondomaine.fr)
       Sender name = Orgue IAMS (ou autre)
   - Authentication → Email Templates : vérifier les liens (redirigent vers Site URL / Redirect URLs).
   - Plan gratuit Brevo : une mention / logo « Brevo » peut apparaître en bas des e-mails ; disparaît sur les offres payantes.
